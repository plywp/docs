# Security Best Practices

Comprehensive security guide for deploying and operating PlyWP securely.

## Security Principles

1. **Defense in Depth** - Multiple layers of security
2. **Least Privilege** - Users/services get minimum required access
3. **Secure Defaults** - Secure out of the box
4. **Encryption Everywhere** - Encrypt data in transit and at rest
5. **Regular Updates** - Keep software patched and current

## Authentication & Authorization

### Strong Passwords

Enforce strong passwords for all users:

```yaml
# Panel configuration
authentication:
  password_policy:
    min_length: 12
    require_uppercase: true
    require_numbers: true
    require_special_chars: true
    expiry_days: 90
```

### Multi-Factor Authentication

Enable MFA for accounts:

1. **TOTP (Time-based One-Time Password)**
   - Use authenticator apps (Google Authenticator, Authy)
   - 30-second time window
   - Backup codes for recovery

2. **Implementation**
   ```javascript
   // Example using speakeasy library
   const secret = speakeasy.generateSecret({
     name: 'PlyWP',
     issuer: 'YourCompany'
   });
   
   // Verify token
   const verified = speakeasy.totp.verify({
     secret: secret.base32,
     encoding: 'base32',
     token: userToken
   });
   ```

### API Token Security

1. **Token Generation**
   - Use cryptographically secure random generation
   - Minimum 32 characters (256 bits)
   - Hash before storage (bcrypt or similar)

2. **Token Rotation**
   - Rotate tokens regularly (monthly)
   - Revoke old tokens immediately
   - Monitor token usage

3. **Scope Limiting**
   ```yaml
   api_key:
     name: "CI/CD Pipeline"
     scopes:
       - sites:read
       - backups:write
     expires_in: 31536000  # 1 year
   ```

## Network Security

### HTTPS/TLS

Always use HTTPS in production:

```nginx
server {
    listen 443 ssl http2;
    server_name plywp.example.com;
    
    ssl_certificate /etc/letsencrypt/live/plywp.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/plywp.example.com/privkey.pem;
    
    # Use modern TLS
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
    
    # HSTS header
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
}
```

### Firewall Configuration

Restrict network access:

```bash
# UFW firewall rules
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Allow HTTPS
sudo ufw allow 443/tcp comment "HTTPS"

# Allow SSH from specific IPs
sudo ufw allow from 203.0.113.0/24 to any port 22 comment "SSH from office"

# Internal Plyorde (restrict to Panel server)
sudo ufw allow from 10.0.1.10 to any port 8081 comment "Panel to Plyorde"

# Deny everything else
sudo ufw enable
```

### VPN/Private Networks

For sensitive deployments, use VPN:

```bash
# Wireguard or OpenVPN
# Keep all Panel-Plyorde communication on private network
# Example Wireguard setup:
[Interface]
PrivateKey = <key>
Address = 10.0.0.1/24
ListenPort = 51820

[Peer]
PublicKey = <peer_key>
AllowedIPs = 10.0.0.2/32
```

## Data Protection

### Encryption at Rest

1. **Database Encryption**
   ```bash
   # PostgreSQL encryption
   # Use pgcrypto extension
   CREATE EXTENSION pgcrypto;
   
   # Encrypt sensitive columns
   ALTER TABLE api_keys 
   ADD COLUMN key_encrypted bytea;
   ```

2. **Backup Encryption**
   ```bash
   # Encrypt backups with gpg
   gpg --symmetric --cipher-algo AES256 backup.tar.gz
   
   # Plyorde backup encryption
   backup:
     encryption:
       enabled: true
       algorithm: AES256
       key_file: /etc/plywp/backup.key
   ```

3. **File System Encryption**
   ```bash
   # LUKS full disk encryption
   sudo cryptsetup luksFormat /dev/sda1
   sudo cryptsetup luksOpen /dev/sda1 plywp-data
   ```

### Encryption in Transit

1. **API Communication**
   - All API calls over HTTPS
   - Certificate pinning for critical connections
   - Mutual TLS for service-to-service

2. **Database Connection**
   ```yaml
   database:
     ssl_mode: require
     ssl_ca: /etc/plywp/ca.crt
   ```

### Secrets Management

Never hardcode secrets. Use secure secret storage:

1. **Environment Variables**
   ```bash
   # Use .env files with restricted permissions
   chmod 600 .env
   
   # Load from secure storage
   export DATABASE_PASSWORD=$(vault kv get -field=password secret/plywp)
   ```

2. **HashiCorp Vault**
   ```bash
   vault auth enable approle
   vault secrets enable -path=secret/ kv
   
   # Store secrets
   vault kv put secret/plywp \
     database_password=... \
     api_secret=... \
     jwt_secret=...
   ```

3. **AWS Secrets Manager**
   ```python
   import boto3
   
   client = boto3.client('secretsmanager')
   secret = client.get_secret_value(SecretId='plywp/db-password')
   password = secret['SecretString']
   ```

## Application Security

### Input Validation

Validate all inputs:

```javascript
// Example validation
const validateSiteName = (name) => {
  if (!name || typeof name !== 'string') {
    throw new Error('Invalid site name');
  }
  
  if (name.length < 3 || name.length > 100) {
    throw new Error('Site name must be 3-100 characters');
  }
  
  if (!/^[a-zA-Z0-9\s\-\.]+$/.test(name)) {
    throw new Error('Site name contains invalid characters');
  }
  
  return name.trim();
};
```

### Output Encoding

Prevent XSS attacks:

```javascript
// React automatically escapes output
<div>{userInput}</div>  // Safe

// Or explicitly encode
const escapeHtml = (text) => {
  const div = document.createElement('div');
  div.textContent = text;
  return div.innerHTML;
};
```

### SQL Injection Prevention

Always use parameterized queries:

```go
// Good - parameterized query
rows, err := db.Query(
  "SELECT * FROM sites WHERE id = ?",
  siteID,
)

// Bad - string concatenation
query := fmt.Sprintf("SELECT * FROM sites WHERE id = '%s'", siteID)
```

### CSRF Protection

Implement CSRF tokens:

```javascript
// Generate token
const csrfToken = generateToken();

// Include in forms
<form>
  <input type="hidden" name="_csrf" value={csrfToken} />
</form>

// Validate on server
if (request.body._csrf !== session.csrfToken) {
  throw new Error('CSRF validation failed');
}
```

### Rate Limiting

Prevent abuse and brute force:

```javascript
// Rate limiting middleware
app.use(rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max: 100,                   // 100 requests
  message: 'Too many requests',
  skip: (req) => req.user?.admin // Skip for admins
}));

// Login-specific rate limiting
loginRouter.use(rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,  // 5 attempts
  skip: (req) => req.user?.admin
}));
```

## Database Security

### PostgreSQL Hardening

1. **Access Control**
   ```sql
   -- Create dedicated database user
   CREATE ROLE plywp WITH LOGIN ENCRYPTED PASSWORD 'secure_password';
   
   -- Grant minimal privileges
   GRANT CONNECT ON DATABASE plywp TO plywp;
   GRANT USAGE ON SCHEMA public TO plywp;
   GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO plywp;
   
   -- Revoke unnecessary privileges
   REVOKE CREATE ON SCHEMA public FROM PUBLIC;
   ```

2. **Audit Logging**
   ```sql
   -- Enable audit logging
   CREATE TABLE audit_log (
     id SERIAL PRIMARY KEY,
     table_name TEXT,
     operation TEXT,
     user_name TEXT,
     old_data JSONB,
     new_data JSONB,
     timestamp TIMESTAMP DEFAULT NOW()
   );
   
   -- Create trigger for tracking changes
   CREATE TRIGGER audit_trigger
   AFTER INSERT OR UPDATE OR DELETE ON sites
   FOR EACH ROW EXECUTE FUNCTION audit_function();
   ```

3. **Connection Security**
   ```ini
   # postgresql.conf
   ssl = on
   ssl_cert_file = '/path/to/server.crt'
   ssl_key_file = '/path/to/server.key'
   
   # pg_hba.conf
   # Require SSL for remote connections
   host    all    all    0.0.0.0/0    md5 sslmode=require
   ```

## File System Security

### Permissions

Set restrictive permissions:

```bash
# Panel application
sudo chmod 755 /opt/plywp/panel
sudo chmod 644 /opt/plywp/panel/*.js
sudo chmod 600 /opt/plywp/panel/.env

# Plyorde application
sudo chmod 755 /opt/plywp/plyorde
sudo chmod 600 /etc/plywp/config.yml
sudo chmod 600 /var/log/plywp/*.log

# Backup storage
sudo chmod 700 /var/backups/plywp
sudo chown -R backup:backup /var/backups/plywp
```

### Directory Permissions

```bash
# Restrict web access to WordPress directories
sudo find /var/www -type f -exec chmod 644 {} \;
sudo find /var/www -type d -exec chmod 755 {} \;

# WordPress writable directories
sudo chmod 775 /var/www/*/wp-content/uploads
sudo chmod 775 /var/www/*/wp-content/plugins
```

## Monitoring & Auditing

### Security Logging

Log all security events:

1. **Login Attempts**
   ```javascript
   // Log all login attempts
   const logLoginAttempt = (username, success, ipAddress) => {
     logger.info('Login attempt', {
       username,
       success,
       ipAddress,
       timestamp: new Date().toISOString()
     });
   };
   ```

2. **API Access**
   ```go
   // Log API requests
   log.Printf(
     "API request: %s %s from %s by %s\n",
     r.Method, r.URL.Path, clientIP, userID,
   )
   ```

3. **Administrative Actions**
   ```javascript
   // Log admin actions
   const logAdminAction = (admin, action, resource, changes) => {
     auditLog.create({
       admin_id: admin.id,
       action,
       resource_type: resource.type,
       resource_id: resource.id,
       changes: JSON.stringify(changes),
       ip_address: request.ip,
       timestamp: new Date()
     });
   };
   ```

## Incident Response

### Incident Response Plan

1. **Detection**
   - Monitor logs for suspicious activity
   - Set up alerting for anomalies
   - Regular security audits

2. **Containment**
   - Isolate affected systems
   - Revoke compromised credentials
   - Collect evidence

3. **Eradication**
   - Remove malware or unauthorized access
   - Patch vulnerabilities
   - Reset credentials

4. **Recovery**
   - Restore from clean backups
   - Verify system integrity
   - Monitor for persistence

5. **Lessons Learned**
   - Document incident
   - Update security procedures
   - Improve monitoring
